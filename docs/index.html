How Derby betting works

If you’ve never bet a horse race, here’s the whole thing in three boxes — everything below it on the page is just applying these ideas to the 2026 field.

Bet types
Win, Place, Show

WIN — horse finishes 1st. PLACE — finishes 1st or 2nd. SHOW — finishes 1st, 2nd, or 3rd. Show pays the smallest, win pays the most. The table below shows the model’s probability for each.

Reading the price
Morning-line odds

Odds like 6.5-1 mean a $1 win bet returns $6.50 in profit if the horse wins. Convert to a probability with 1 / (6.5 + 1) = 13.3% — that’s the “market implied” chance the bookies are offering.

Finding the edge
BET, FAIR, FADE

Compare the model’s win% to the market’s implied%. BET if model >15% above market. FADE if >15% below. Otherwise FAIR.

Reality check: Churchill takes ~17% out of the win pool before paying. To actually break even you need closer to +20%, not 15%. After takeout, only the biggest gaps are real edge.

Pari-mutuel reality
Morning line ≠ closing odds

The morning line is the track oddsmaker’s guess at where the public will bet, not the actual market. By post, Renegade (4-1 ML) will likely be 7-2 or shorter; Further Ado will move too. We’re comparing the model to a prediction of the market, not the closing tote. Real edge gets evaluated at the bell.

The takeout problem
Churchill keeps ~17%

Churchill removes ~17% from the win pool and ~22% from exotics before payouts. To break even, the model must be roughly +20% above market, not +15%. After takeout, only Further Ado (+15.3 net) and Chief Wallabee (+13.0 net) clear cleanly. The Puma and Emerging Market are washes; Six Speed is a sprinkle.

The backtest fails its own null test
p = 0.87 against permuted labels

We re-ran the same 5,000-Dirichlet search on 2,000 random shuffles of who actually won, on Burla. Under shuffled labels, the search hits the published 22/40 score or higher in 87.4% of permutations; the median null is 30/40. The headline is below the noise floor of its own search procedure. What the model doesn’t know
Beyer alone is 1990s

No pace figures (separate from speed), no trip notes, no Ragozin/Thoro-Graph sheets, no recent works, no layoff days, no current jockey-trainer combo strike rate, no weather, no live-odds movement. Beyer + run style + dosage is a starting point, not a complete card. If you have access to those, weight these calls accordingly.

Methodology audit

Where this model is fragile, and what we ran on Burla to prove it.

A skeptical data scientist would tear into the pipeline behind this site. Rather than wait for someone else to do it, we did it ourselves — including a real permutation test on Burla — and put the receipts here. The 1-trillion-sim headline doesn’t fix bad inputs.

1. The 22/40 backtest is below its own null floor

The site previously trumpeted “22/40 backtest hits, best of 5,000 Dirichlet-sampled weight combinations on 2022–25.” A permutation test on Burla asks the obvious question: if you shuffle who actually won and re-run the same 5,000-combo search, what does the best score look like under random labels? If the model is real, shuffled labels should produce much lower scores. We ran 2,000 permutations, each searching 5,000 Dirichlet weight vectors, in 13.8 seconds across the cluster.

There IS a small real-label edge over a random pick (+7 points), so the underlying weighted-feature framework isn’t worthless — but anyone calling 22/40 “empirically validated” is confusing “the maximum of a big random search” with “a calibrated estimate.”

2. Five upstream data frauds the simulation can’t fix

No amount of Burla compute fixes garbage inputs. From a code review of derby_scraper.py, derby_features.py, derby_model.py, and derby_sensitivity.py:

  1. Beyer figures for non-winners are synthesized from finish position. derby_scraper.py line 194: beyer = winner_beyer - 3 × (finish_pos - 1). The “historical Beyer distribution” the model trains on is invented. So is the dosage index. So when the site says “Further Ado’s 106 Beyer leads the field,” the comparison universe was generated by arithmetic, not measurement.
  2. The “Burla scrape” never wrote anything. The script calls remote_parallel_map(scrape_hrn_year, years), ignores the result, and writes the hard-coded FALLBACK_DATA DataFrame. The “26 page-fetches in parallel via Burla bypass IP blocks” pitch is theatrical — the data is hard-coded in Python.
  3. The 5,000-combo sensitivity backtest runs against hand-typed feature scores. derby_sensitivity.py defines BACKTEST_FIELDS with literal entries like dict(name="Rich Strike", beyer_norm=2.7, …). These values were chosen by hand, not computed by the same feature pipeline. The “best weights” optimize against arithmetic, not data. The backtest only includes 4–5 horses per year (the top finishers), not the 20-horse field.
  4. The ML training feature set leaks the answer. FEATURE_COLS includes implied_prob = 1 / (odds + 1). Since morning-line odds embed the market’s prediction of who will win, the classifier mostly relearns the market. That’s why the “best log-loss 0.378” sounds impressive but isn’t meaningfully better than just betting the favorites.
  5. The ML output is essentially uniform garbage. From model_results.json → horse_ml_probs: Robusta (50-1) gets 10.6%, Further Ado gets 10.5%, and Renegade (the actual 4-1 morning-line favorite) gets 3.6%. The ML “ensemble” isn’t learning — an n=88 training set with 4 winners and a leaky implied-prob feature produces near-random predictions. The published win% lives mostly in the hand-tuned 70%-weighted-score component, not the ML 30%.

3. The hand-coded “features” are opinions

EXPERT_SCORES, PACE_FIT, and POST_DRAW_ADJUSTMENTS in derby_features.py are hard-coded Python dicts dressed as features. The comments claim “aggregated from Aces & Races, CBS, Yahoo, SportsLine” with no aggregation code, no source URLs, no timestamp. They’re one person’s priors typed into a constant.

4. The 1-trillion-sim count is honest, but doesn’t add information

The Monte Carlo really did run 50,000 workers x 20,000,000 sims = 1T total in 18.3 minutes on Burla. That’s legit cloud compute. But the Monte Carlo just adds Gumbel noise to a fixed score vector and counts orderings — variance reduction past about 100,000 sims is irrelevant because the inputs (those scores) come from the broken pipeline above. 1 trillion samples gets you 6-decimal precision on a quantity whose true value is unknowable.

What this site is and isn’t

What we trust: the 2026 horse stats are face-valid (Beyer, run style, jockey/trainer Derby wins, sire pedigree) and the relative ranking — Further Ado > Chief Wallabee > The Puma > rest — tracks how a thoughtful handicapper would order these horses on the same inputs. The 70/30 weighted blend is mostly a reasonable hand-tuned scorecard. As a ranking, it’s a defensible first pass.

What we don’t trust: the specific probabilities (31.26%, 26.43%, …) to four decimals as if they were calibrated; the “BET / FAIR / FADE” classifications that compare those probabilities to morning-line implied%; the “empirically validated weights” framing; the implication that more compute equals more truth. This is a ranked handicapper’s sheet with a Burla flex on top, not a published Bayesian probability model. Treat it accordingly. How it ran on Burla

Burla is a Python library for parallel cloud compute. You write a function, call remote_parallel_map, and it fans out across a cluster instantly. No Docker, no Kubernetes, no orchestration glue.

For this pipeline, three separate Burla calls did all the heavy lifting: 164 ML model configurations trained in parallel, 5,000 weight combinations back-tested against 2022–2025 results in 7 seconds, and 1,000,000,000,000 race simulations run as 50,000 parallel Burla workers in 18.3 minutes.

50,000 concurrent Burla workers at peak during Monte Carlo, each running 20,000,000 race simulations. 5,000 weight-combo workers ran simultaneously during sensitivity analysis — all results back in 7 seconds.

Full source code on GitHub. Burla docs at docs.burla.dev. 

# Test 5,000 weight combinations across all 10 factors.
# Back-test each against 2022-2025 Derby results.
# Returns in 7 seconds on Burla (5,000 parallel workers).
from burla import remote_parallel_map
import numpy as np

def backtest_weights(weights_list, factors, backtest_fields):
    weights = np.array(weights_list)
    total_score = 0
    for year, horses in backtest_fields.items():
        scores = [(h["name"], sum(weights[i] * h[f]
                   for i, f in enumerate(factors)))
                  for h in horses]
        scores.sort(key=lambda x: -x[1])
        actual = next(h["name"] for h in horses if h["is_winner"])
        rank = next(i for i, (n, _) in enumerate(scores) if n == actual)
        total_score += [10, 5, 2, 1, 0][min(rank, 4)]
    return {"weights": weights_list, "score": total_score}

# Dirichlet sample — uniform over the probability simplex
combos = np.random.dirichlet(np.ones(10), size=5000).tolist()
args = [(c, FACTORS, BACKTEST_FIELDS) for c in combos]

# 5,000 workers run simultaneously → results in 7 seconds
results = remote_parallel_map(backtest_weights, args, grow=True)
best = max(results, key=lambda r: r["score"])
# → jockey_quality: 32.5%, pedigree_dist: 16.6%, post_itm: 13.4%

# Each worker runs 20,000,000 sims, returns position tallies.
from burla import remote_parallel_map
import numpy as np

def simulate_race_batch(scores, n_sims, seed):
    rng = np.random.default_rng(seed)
    n = len(scores)
    counts = np.zeros((n, 4), dtype=np.int64)
    for _ in range(n_sims):
        noise  = rng.normal(0, 1.8, n)       # calibrated upset rate
        noisy  = np.array(scores) + noise
        exp_s  = np.exp(noisy - noisy.max())
        probs  = exp_s / exp_s.sum()
        order  = rng.choice(n, size=4, replace=False, p=probs)
        for rank, idx in enumerate(order):
            counts[idx][rank] += 1
    return {"counts": counts.tolist()}

# 50,000 workers × 20,000,000 sims = 1,000,000,000,000 total in ~18 minutes
args = [(log_probs, 10_000, seed) for seed in range(100)]
results = remote_parallel_map(simulate_race_batch, args, grow=True)
total = np.sum([r["counts"] for r in results], axis=0)

What’s the point?

The real result isn’t the win probability. It’s the time it took to build a rigorous model from scratch.

If you ran all three compute steps locally — sequentially, with Python’s GIL limiting CPU threads — the sensitivity analysis alone would take ~20 minutes. The full pipeline would take the better part of an afternoon.

With Burla, the same pipeline runs in under 3 minutes of actual compute time. That changes how you iterate: you can test 5,000 weight hypotheses, not 50. You can train 164 model configurations before picking a winner, not 5. You can run a million simulations, not ten thousand.

The sensitivity analysis found that jockey quality carries 32.5% of the predictive weight — more than double what a conventional Beyer-first model would assign. That insight only emerged because we could afford to test 5,000 combinations.
